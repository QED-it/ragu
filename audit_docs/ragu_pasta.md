## Background for Understanding `ragu_pasta`

### What the crate does

`ragu_pasta` is the concrete instantiation of Ragu's abstract traits for the **Pasta curve cycle** (Pallas and Vesta). It provides the `Pasta` struct implementing the `Cycle` trait, pre-computed generator points for Pedersen commitments, and Poseidon hash parameters for both fields. This is the only crate that binds the protocol to a specific set of elliptic curves.

It is used exclusively as a **dev-dependency** by other crates for testing, and as the runtime parameter source for actual proof generation.

---

### 1. The Pasta curve cycle

Pallas and Vesta are a cycle of two elliptic curves designed for recursive proof composition:

- **Pallas** (Ep): base field Fp, scalar field Fq
- **Vesta** (Eq): base field Fq, scalar field Fp

The defining property: each curve's scalar field is the other's base field. This means arithmetic over Fp is "native" in a Pallas-based circuit, and Vesta point coordinates (which live in Fq) are also native. Conversely, a Vesta-based circuit can natively manipulate Pallas point coordinates.

The `Pasta` struct implements the `Cycle` trait (from `ragu_arithmetic`) with:
- `CircuitField = Fp`, `ScalarField = Fq`
- `HostCurve = EqAffine` (Vesta), `NestedCurve = EpAffine` (Pallas)

This mapping determines which curve's commitments are "native" (host) vs "nested" for the recursive verifier circuit.

See [Protocol Overview](https://tachyon.z.cash/ragu/protocol/index.html): "Realized using the Pasta curve cycle designed for and deployed in Zcash's Orchard protocol." Also [Cycles of Elliptic Curves](https://tachyon.z.cash/ragu/protocol/index.html#cycles).

---

### 2. Fixed generators

The protocol requires a public set of group generators **G** = (G₀, G₁, ..., G_{n-1}) and a blinding generator H for Pedersen vector commitments. These are generated deterministically via hash-to-curve using the domain separator `"Ragu-Parameters"`.

- **`PallasGenerators`**: 2^13 = 8192 points on Pallas (EpAffine), plus one blinding point H
- **`VestaGenerators`**: 2^13 = 8192 points on Vesta (EqAffine), plus one blinding point H

Both implement `FixedGenerators<C>` (from `ragu_arithmetic`), which provides `g()` → `&[C]` and `h()` → `&C`.

The generator count (8192) appears in the [Analysis](https://tachyon.z.cash/ragu/protocol/analysis.html): "Multi-scalar multiplication size: 8192 group elements."

These generators are what the MSM (`ragu_arithmetic::mul()`) multiplies against when committing polynomials:

> "We commit the polynomial using Pedersen commitment over its coefficient: F ← dot(**f**, **G**)"
> — [Bulletproofs IPA](https://tachyon.z.cash/ragu/protocol/prelim/bulletproofs.html)

---

### 3. The `baked()` feature

Generator computation via hash-to-curve is expensive at runtime. The `baked` feature pre-computes all generators at build time and embeds them as a binary blob (`pasta_parameters.bin`) in the compiled binary.

- **`build.rs`** — generates the binary blob during `cargo build`
- **`Pasta::baked()`** — deserializes from the blob via `lazy_static!`, returning `&'static PastaParams`
- **`Pasta::generate()`** — computes generators at runtime (slow, used when `baked` is off)

The `baked()` path is the normal production path. Tests in other crates use `Pasta::baked()` to get parameters without re-computing.

---

### 4. Poseidon parameters

Two complete Poseidon parameter sets are embedded:

- **`PoseidonFp`** — implements `PoseidonPermutation<Fp>` (for Pallas base field)
- **`PoseidonFq`** — implements `PoseidonPermutation<Fq>` (for Vesta base field / Pallas scalar field)

Both use identical structure:
- T = 5 (state width)
- RATE = 4 (absorption rate)
- FULL_ROUNDS = 8 (4 before + 4 after partial rounds)
- PARTIAL_ROUNDS = 56
- ALPHA = 5 (S-box exponent: x^5)
- 64 round constant arrays, each with 5 field elements
- 5×5 MDS matrix

Constants are generated by the `daira/pasta-hadeshash` script and embedded as hex literals via the `fp!`/`fq!` macros (which use `repr256!` from `ragu_macros`).

Poseidon is used as a [random oracle](https://tachyon.z.cash/ragu/protocol/prelim/assumptions.html) for Fiat-Shamir throughout the protocol, and appears in the [transcript](https://tachyon.z.cash/ragu/protocol/prelim/transcript.html) construction.

---

### 5. Re-exports

The crate re-exports key types from `pasta_curves` for convenience:
- `Ep`, `EpAffine` — Pallas curve and affine points
- `Eq`, `EqAffine` — Vesta curve and affine points
- `Fp` — Pallas base field (= Vesta scalar field)
- `Fq` — Pallas scalar field (= Vesta base field)

And provides helper macros:
- `fp!(hex)` — create Fp from hex literal
- `fq!(hex)` — create Fq from hex literal

---

### How it maps to the protocol

```
Curve cycle for recursion                ← Pasta struct (Cycle trait)
Pedersen vector commitment generators    ← PallasGenerators, VestaGenerators
Polynomial commitment F = <f, G>         ← FixedGenerators::g()
IPA blinding                             ← FixedGenerators::h()
Fiat-Shamir transcript hashing           ← PoseidonFp, PoseidonFq
Random oracle assumption                 ← Poseidon with T=5, α=5
Pre-computed parameters for production   ← baked() feature
```

### Reading order

1. **`pasta_common.rs`** — `PastaParams`, `PallasGenerators`, `VestaGenerators`, hash-to-curve generation
2. **`src/lib.rs`** — `Pasta` struct, `Cycle` impl, `baked()` / `generate()`, re-exports
3. **`src/poseidon_fp.rs`** — Poseidon round constants and MDS for Fp
4. **`src/poseidon_fq.rs`** — Poseidon round constants and MDS for Fq
5. **`src/macros.rs`** — `fp!` / `fq!` helper macros
6. **`build.rs`** — binary blob generation for baked feature
